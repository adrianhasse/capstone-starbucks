Starbucks Capstone Project

Adrian Hasse
Jan 9 · 6 min read

Project Definition

This project is the final capstone projects of the Data Scientist Nanodegree of Udacity provided by Starbucks. The project includes simulated data that mimics customer behavior on the Starbucks rewards mobile app. The data includes information about potential offers, customers and transactions. This allows further analysis of a specific customer group in particular how they interact with offers and incentives.

The data is made available in 3 separate json files:

  • portfolio.json — contains all potential offers of Starbucks
  • profile.json — contains further information of the participating customers
  • transcript.json — contains interactions between Starbucks and customers

There are three types of offers referring to either an advertisement, a discount or a bogo (buy one get one) offer. For each offer, except the advertisement, the customer has to spent a certain amount to get the benefit which is either a discount or an additional beverage. The profiles of each customer provides additional information about the customers like gender, income or when they downloaded the app. The largest data set is called transcript and contains all transactions between the customers and Starbucks including the corresponding time, date and amount as well as their interactions with offers. Whenever a customer receives, views or completes an offer the date, time and offer id is stored.

The question of how likely it is that a particular offer will be completed is now further analyzed. In other words, when a customer receives an offer, how likely is it that it will be completed and he or she receives either a discount or an additional beverage.

This is a rather difficult problem to solve because the data contains a fair amount of noise. For example is it possible that a customer completes an offer without even knowing about the offer. This is the case when no information channel like an app or email reaches the customer. Thus, sending out an offer and completing the same one have often no causal relation.

The above formulated question is answered with a regression model since it is more flexible and also delivers more information concerning the likeliness of an offer to be completed. Therefore, a random forest is used which generalizes quite well and also offers the possibility to reproduce non-linear relationships.

To measure the performance of the regression model the mean squared error is used. This measure has the advantage that it is quite performant for a regression and since there are by definition no outliers in the data, its biggest disadvantage is of the table.

Since the data is simulated and contains quite some noise a fairly accurate model is expected, especially considering the small amount of features. Nevertheless the model should be good enough to give an advantage in practice.

Analysis

There are in total 10 offers which refer to either an advertisement, a discount or a bogo (buy one get one) offer. Each offer can be distributed via web, email, mobile or social channels although not all channels are available for each offer. Each offer remains valid for a certain period of time, which is not more than 10 days.

The data contains 17000 customers of all ages (from 18 till 100) with a reasonable distribution. Also, the income of over 87 % of the customers is available. The pictures below show the age and income distribution.

In total the data contains 306534 events which refer to ‘offer received’, ‘offer viewed’, ‘offer completed’ and ‘transaction’. Furthermore, a customer can receive several different offers simultaneously. This means they may overlap in time which has to be consider for further data preparation. In addition, offers that are ‘informational’, which means that they are just advertisements, cannot be completed.

Most transactions of the customers are between 3 and 10 units of money. This seems plausible assuming that the currency is either Dollar, Euro or Pound. The distribution of income and age also fits the assumption that the simulated data mimics a society using these currencies.

Methodology

Since the data is simulated not much data cleaning is necessary. The following histogram shows one of the exceptions. If an age of a customer is not available the age of 118 is assumed. This was corrected by replacing it with the mean.

In general, missing values refer mostly to categorical variables like ‘gender’. They were treated as a separated category.

In order to find out which offers were completed , one has to iterate over all events. This is also helpful for further feature engineering. The data set was therefore reduced to 61042 data points, all referring to offers of the type ‘bogo’ (buy one get one) and ‘discount’. From these offers 33101 were completed.

I calculated 3 additional features:

  • The average amount spent on each transaction,
  • the average amount spent over time and
  • the average time taken to recognize an offer.

To prevent data leakage, all three features always incorporate the date and time of the corresponding offer. This means, for example, that only transactions that took place before the date and time of the offer were used to calculate the average amount spent on each transaction of a particular offer.

The final data set containing 61042 data points was split into a training set, a test set and a validation set. The training set contains 75 % of the data points, the test set 18.75 % and the validation set 6.25 %. This gives us enough data point for training the model while still leaving more than 3,800 data point for the final validating process and robustness test.

Choosing a random forest as regression model, cross-validation was used to tune hyper-parameters. Especially the minimum number of samples in a leaf turned out to increase the model quality and performance. This parameter prevents the tree from further splitting up a tree by demanding a certain minimum amount of remaining data points. This prevents the model from over-fitting but also increase the calculation speed since less splittings have to be performed.

Using cross-validation to further adjust the random forest model a mean squared error of around 35 % was achieved in the final validation set. This is not no bad considering that there were not to many features available. To get a better impression which features were the most important, I feature analysis was produced.

Results

The final outcome of the model lead to a mean squared error of 36 % for the test set and 35 % for the validation set. Although a cross-validation was performed, the model seems to be very robust, since the mean squared error is not significantly different.

To gather further information concerning the features, an additional feature importance analysis was performed. It turned out, that the the three engineered feature increased the model quality drastically shown the the bar chart below.

In particular the average amount spent for each transaction added improved the quality a lot.

The final results meet expectations, especially considering the discussed noise contained in the data. The model could certainly be used for a better predicting if an offer will be successful.

Conclusion

The project focuses on an important aspect of each retail company: How to set the right incentives to increase the willingness to spend of each customers, while at the same time increasing or at least not decreasing customer loyalty. This can also be transferred to many other problems and industries. Thinking about environmental problems the government could also set certain incentives to decrease pollution. Thus I found the project very interesting and also challenging.

Especially the need to iterate over all events had to be implemented carefully since a lot of logic was involved there.

Especially the feature importance analysis unveiled the importance of feature engineering. Increased efforts in this topic could further increase the final model performance.
